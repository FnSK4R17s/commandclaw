# commandclaw-mcp — the MCP gateway

> Agents never hold real credentials. They hold phantom tokens that expire within the hour and only work through this proxy.

## Why it exists

The default way to give an agent a tool is: hand the agent an API key and let it call the API. This is catastrophic for credentialed tools. A prompt injection, a misrouted context dump, or a leaky tool response, and the key is live on the internet. The median time from credential leak to first malicious use is around five minutes.

`commandclaw-mcp` separates the agent from its credentials with a proxy architecture:

```
Agent ──(phantom token)──► commandclaw-mcp ──(real credential)──► External MCP server
```

The agent holds a short-lived, rotating phantom token. The gateway holds the real credential. Even if the phantom token leaks, it expires within the hour and only works through the gateway — which can enforce RBAC, rate limits, and audit logging on every call.

This is one of three proxies in the ecosystem. The [LLM gateway](commandclaw-gateway.md) does the same pattern for model providers. The [memory service](commandclaw-memory.md) does it for the wiki.

## What it does

| Capability | Detail |
|---|---|
| **Phantom tokens** | Per-agent tokens with hourly rotation. Short-lived, bounded-scope, revocable. |
| **Credential isolation** | Real API keys live in the gateway's encrypted store, never reach the agent. |
| **RBAC** | Two-layer: Cerbos policies + local session-scope checks. Per-agent tool allowlists. |
| **Session pooling** | Reuses upstream MCP sessions across agent calls. Amortizes handshake cost. |
| **Upstream proxy** | Fronts arbitrary MCP servers (filesystem, web, GitHub, Slack, custom). Agent sees one endpoint. |
| **Observability** | OpenTelemetry traces, Prometheus metrics, structlog structured logs. |
| **Encrypted config** | Secrets in `~/.commandclaw/mcp.json` encrypted with a per-install seed (generated by `make setup`). |

## Tech stack and why

| Choice | Reason |
|---|---|
| **Python 3.12+** | Matches the MCP SDK's support baseline; newer asyncio improvements (`TaskGroup`, `asyncio.timeout`) cleaner than 3.11 shims. |
| **FastAPI** | Async-first HTTP, automatic OpenAPI, clean dependency injection for auth middleware. |
| **FastMCP** | Thin FastAPI-friendly MCP library. Speaks MCP without reinventing the protocol. |
| **Redis (with hiredis)** | Token storage, rate limit counters, session pool state. In-memory speed, persists for restarts, hiredis for protocol speed. |
| **Cerbos** | Policy-as-code RBAC engine. Policies live in `policies/` — versioned, diffable, testable. Runs as a sidecar. |
| **cryptography** | Config encryption. Fernet + per-install seed. |
| **argon2-cffi** | Phantom token hashing. Argon2id is the current best-practice memory-hard KDF. |
| **httpx** | Async HTTP to upstream MCP servers. Connection pooling, HTTP/2. |
| **OpenTelemetry** | Standardized tracing across MCP calls, HTTP, Redis. Exports OTLP, so any backend works. |
| **Prometheus client** | Request counts, latencies, RBAC denials, token rotations. |
| **structlog** | JSON-structured logs by default. Key-value pairs instead of printf. |
| **mypy + respx + fakeredis** | Type-checked code, HTTP mocks for tests, fake Redis to avoid a test-time dependency. |

## Internal architecture

```
src/commandclaw_mcp/
├── gateway/           HTTP entry — /mcp, /admin, /health
├── auth/              Phantom token issue, rotate, validate
├── rbac/              Cerbos client, policy evaluation
├── security/          Encryption seed, token hashing (argon2id)
├── session/           Upstream MCP session pool, lifecycle
├── middleware/        FastAPI middleware — auth, tracing, rate limit
└── observability/     OTEL setup, Prometheus metrics, structlog config

policies/              Cerbos policies (versioned, diffable)
examples/              Sample configs and test servers
```

### Request flow

```
Agent HTTP request
    │  Authorization: Bearer <phantom-token>
    ▼
Auth middleware  ──► look up hash in Redis ──► attach agent identity to request
    │
    ▼
RBAC middleware  ──► Cerbos: "can <agent> call <tool>?" ──► deny or pass
    │
    ▼
Session pool     ──► acquire upstream MCP session (create if needed)
    │
    ▼
Proxy call       ──► forward MCP request with real credentials
    │
    ◄── response
    │
    ▼
Observability    ──► emit trace span, Prometheus counter, structured log
    │
    ▼
Agent HTTP response
```

Every layer can fail fast with a clear error. Nothing about the upstream server leaks to the agent except what RBAC permits.

### Phantom token lifecycle

1. Agent enrolls → gateway issues `phantom-<random>` token, stores Argon2id hash + TTL in Redis.
2. Agent includes token in every MCP request.
3. Gateway rotates automatically at the configured interval (default: hourly); old token valid during grace period.
4. Agent transparently gets the new token via the enrollment channel.
5. Leaked tokens expire on their own; admin can force-revoke instantly.

## External interfaces

| Interface | Direction | Used for |
|---|---|---|
| **HTTP (MCP wire format)** | [runtime](commandclaw.md) → gateway | Tool calls. |
| **HTTP/stdio MCP** | gateway → upstream servers | Executes the actual tool call. |
| **Redis** | gateway ↔ Redis | Token store, rate limits, session state. |
| **gRPC** | gateway ↔ Cerbos | Policy evaluation. |
| **OTLP** | gateway → trace backend | Distributed traces. |
| **`/metrics`** | gateway → Prometheus | Scrape endpoint. |

## Deployment shape

```yaml
# docker-compose.yml
services:
  commandclaw-mcp:     # this repo — FastAPI + FastMCP
    ports: ["8420:8420"]
  redis:               # token store, session pool, rate limits
  cerbos:              # policy engine
```

Config at `~/.commandclaw/mcp.json` (outside the repo, never in git). `make setup` generates a starter config with a random encryption seed. `make up` brings the full stack.

```bash
make setup        # one-time: generate config + seed
make up           # bring up gateway + Redis + Cerbos
make health       # check /health + /ready
make dev          # run gateway locally, deps in Docker
```

## Why an MCP gateway over alternatives

| Alternative | Why we didn't |
|---|---|
| **Hand agent the real keys** | One prompt injection = compromised credential. Five-minute median to abuse. |
| **Per-request OAuth** | Many MCP tools don't support per-request OAuth, and even when they do, handshake cost kills UX. |
| **Vault + envelope encryption** | Good for at-rest keys, doesn't help when the agent decrypts them in memory to call. |
| **A monolithic "agent framework" that bundles tool auth** | Couples credential policy to framework version. Proxy is the clean boundary. |

The proxy wins because it is the *only* layer that can enforce runtime policy — token rotation, RBAC, rate limits, audit — without the agent's cooperation.
