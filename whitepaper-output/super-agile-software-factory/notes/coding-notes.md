# super-agile-software-factory — Coding Notes

## Questions

### Q1: What makes a long-horizon coding loop compound correctness rather than accumulate error, and what harness-level design choices enable this?

The canonical failure mode of iterative LLM-based coding is error accumulation: each generation step conditions on previous outputs, so a single wrong assumption propagates and amplifies through the chain. Academic research (arXiv:2604.11978, 2025) [CODE-9] characterises this formally — even a modest per-step error rate, compounded across tens of interdependent actions, drives task success toward near-zero at long horizons. The inflection that made the software factory paradigm viable was identified precisely when this dynamic reversed. According to StrongDM's factory documentation [CODE-1], Claude 3.5 (October 2024 revision) was the first model to demonstrate "compounding correctness rather than error" in long-horizon agentic coding — meaning successive iterations converged on functional solutions rather than diverging from them. Simon Willison's contemporaneous analysis [CODE-2] pins a broader industry inflection at November 2025, when Claude Opus 4.5 and GPT 5.2 reached sufficient reliability for complex autonomous workflows.

Several harness-level design choices are responsible for this property rather than model capability alone.

**Errors as recoverable signals, not fatal exits.** Attractor's coding agent loop specification [CODE-3] mandates that tool-level failures — file not found, edit conflict, shell timeout — are caught and surfaced to the model as structured error results rather than crashing the session. The loop therefore treats every failure as a teachable event rather than an abort condition.

**Loop detection with steering injection.** The Attractor loop spec monitors the last 10 tool calls for repeating patterns. When a cycle is detected, it injects an explicit steering message into the conversation: "the last N tool calls follow a repeating pattern. Try a different approach." This interrupts stuck attractor states without terminating the session, preserving accumulated context while forcing divergence from the failing trajectory.

**Provider-specific tool alignment over universalism.** The spec explicitly avoids forcing a single tool interface across model families. OpenAI models receive `apply_patch` (v4a format); Anthropic models receive `edit_file` (old_string/new_string); Gemini aligns with gemini-cli conventions. The rationale is that "each model family works best with its native agent's tools and system prompts" [CODE-3] — respecting training-data alignment rather than imposing a lowest-common-denominator abstraction that degrades output quality.

**Output truncation with explicit markers.** Long-horizon loops blow up context windows. Attractor truncates tool output by character then line limits, but inserts an explicit warning: "[WARNING: Tool output was truncated. N characters removed...]". The full output remains available in event streams. This prevents the model from reasoning over silently incomplete data.

**Subagent decomposition with scoped histories.** The loop supports spawning child sessions for parallel work. Subagents share the parent's filesystem but maintain independent conversation histories. Depth limits prevent recursive spawning. This enables parallel exploration of solution branches while keeping each subagent's context narrow enough to remain reliable — divide-and-conquer applied to context window management.

**Scenario-based validation as the outer harness.** StrongDM replaces traditional boolean test suites with scenario testing [CODE-1]: end-to-end user stories, stored outside the codebase as holdout sets the agent cannot see during development, evaluated by "satisfaction" — the fraction of observed trajectories likely meeting user needs. This closes the feedback loop correctly: because both code and tests might otherwise be generated by the same agent, scenario holdouts serve as a ground-truth signal the agent cannot overfit to.

**Digital Twin Universe for safe feedback at scale.** The Digital Twin Universe (DTU) [CODE-1] provides high-fidelity behavioral clones of external services (Okta, Jira, Slack, Google Docs). This enables thousands of scenario executions per hour without rate limits or costs, and allows failure-mode injection that would be dangerous or impossible against production APIs. The volume of feedback signals matters: at low iteration rates, stochastic errors are not smoothed; at high rates, the statistical signal overwhelms noise, which is the mechanical reason correctness compounds rather than accumulates error.

### Q2: What are the supporting techniques — Gene Transfusion, Semports, Pyramid Summaries — as described by StrongDM, and how do they generalise to any factory?

StrongDM's factory documentation [CODE-4, CODE-5, CODE-6] describes three named techniques that address distinct failure modes of autonomous long-horizon development: pattern reuse across contexts, cross-language synchronisation, and context-window management under finite attention.

**Gene Transfusion [CODE-4]**

The technique addresses the problem of pattern reuse across diverse codebases without shared authorship or shared library infrastructure. Its central claim: "a solution paired with a good reference can be reproduced in new contexts." The factory documentation describes a five-step flow. First, an exemplar is identified — a working implementation, drawn from inside or outside the organisation's code estate. Second, the agent extracts the structural invariants and edge-case handling from the exemplar. Third, it synthesises an equivalent implementation adapted to the target context: different language, different architecture, different local constraints. Fourth, behavioral tests confirm functional equivalence between the original and the new implementation. Fifth, the reproduced pattern is catalogued for future transfusions elsewhere. Three propagation modes are enumerated: cross-language application, direct inlining without abstraction overhead, and library embodiment as a formalised dependency.

The generalisation principle is direct: any factory maintaining multiple services, multiple language targets, or multiple customer deployments faces the same combinatorial problem — good solutions discovered in one context need to reach others without human intermediation. Gene Transfusion makes this a pipeline primitive rather than a one-off manual task. In practice, the technique also reduces the cognitive load problem of specifying novel solutions from scratch: giving the agent a concrete exemplar dramatically narrows the solution space and anchors the synthesis to a verified behavioral baseline.

**Semport (Semantic Port) [CODE-5]**

Semport addresses cross-language synchronisation: the problem of benefiting from upstream library development without being constrained by upstream technology choices. The name contracts "semantic port" — the goal is to transfer meaning, not syntax. Three port types are distinguished. One-time ports migrate a library to another language for independent maintenance. Ongoing ports continuously synchronise upstream changes, automatically evaluating whether new commits apply to the target implementation. Adaptive ports reshape APIs to match internal conventions while preserving semantics.

StrongDM's concrete example: they monitor the OpenAI Python agents repository daily. When updates appear, their system evaluates whether the changes apply to their Go implementation. During translation, the coding agent identifies bugs that exist in the Python original but are "not expressible in Go" — type-system or concurrency-model constraints surface implementation errors in the upstream that the port makes explicit. Attractor then "ledgers the fix, runs the tests, tags the release" [CODE-5] — the full release cycle is automated.

The generalisation: any factory working across language boundaries or importing ideas from open-source ecosystems without adopting their stack can treat Semport as a continuous integration primitive. The pattern also applies within a single language across framework versions: an ongoing port from one framework revision to another is structurally identical. The key insight is that semantic equivalence is verifiable by behavioral tests, making the automation tractable — the agent does not need to prove logical equivalence, only satisfy a scenario harness.

**Pyramid Summaries [CODE-6]**

Pyramid Summaries address context-window exhaustion: the fundamental finite-attention constraint on any agent reasoning over large codebases or large collections of artifacts. The technique takes its name from multi-resolution image formats (Pyramid TIFF) and map tile systems. The core operation is reversible summarisation at multiple levels of compression. The factory documentation illustrates with a prompt sequence: "Summarise this bug report in 2 words. Now 4. Now 8. Now 16." Each level doubles detail while preserving essential meaning.

The operative workflow combines Pyramid Summaries with a MapReduce pattern. Map: generate summaries for each artifact in parallel at the most compressed level. Cluster: group related items using compressed representations. Reduce: synthesise insights across clusters, expanding detail selectively only where signal demands it. The analogy offered is executive decision-making: starting with a high-level organisational view, drilling into a department, then a team, then an individual — expanding only where needed.

The generalisation: any factory operating on codebases larger than a single context window — which is to say, any production-scale factory — needs a mechanism to enumerate many items without displacing actionable context. Pyramid Summaries make enumeration and selective expansion first-class operations rather than ad-hoc prompt engineering. The technique also composes with Gene Transfusion: identifying candidate exemplars across a large estate is a map-then-expand operation, surveying at compressed level to find patterns, then expanding to full detail before extraction.

### Q3: What is the non-interactive coding agent pattern (Attractor-style) and how does it differ from interactive CLI agents like Claude Code, Aider, and Cursor's agent mode?

**The Attractor pattern** [CODE-3, CODE-7] is defined by a specific execution contract: the agent receives a fully specified task — natural language specifications, scenarios, tool access — and operates end-to-end without prompting the user for decisions, clarification, or approval. Attractor's architecture is a directed graph where nodes represent phases (Implement, Identify, Optimise, Validate) and edges are natural-language transition conditions evaluated by the LLM itself. Execution is deterministic from identical inputs, observable at every node transition via an event stream, resumable from any checkpoint, and composable with other pipeline graphs. The repository contains no executable code — only three Natural Language Specification (NLSpec) markdown files describing the agent, the coding loop, and the unified LLM client. The specification is implemented by feeding those documents to the coding agent of choice. By early 2026, 18+ community implementations existed across Rust, Python, TypeScript, Java, and C#.

The non-interactive contract has specific consequences. There is no mechanism for the agent to ask "did you mean X or Y?" mid-task. This places the entire specification burden on the task definition phase — the Shift Work technique [CODE-1] describes this as dividing labour: humans write complete specifications and scenarios upfront, then agents execute end-to-end. If the specification is underspecified, the agent produces a coherent but wrong result; there is no human-in-the-loop to catch the divergence mid-run. This is why scenario testing with holdout sets is not optional but load-bearing: it is the primary mechanism for discovering specification errors.

**Interactive CLI agents — Claude Code, Aider** — are designed around the opposite contract. The human is in the loop at every significant decision point. Claude Code (Anthropic) operates in terminal, VS Code, JetBrains, and desktop contexts, executing multi-step tasks but pausing for human confirmation on destructive operations, permission checks, and ambiguous intent. It can be run headlessly (`--non-interactive` flag) for scripted use in CI, but its interaction model is fundamentally designed for a developer sitting at a terminal. Aider treats Git as a first-class citizen, committing after every meaningful change with descriptive messages — the human reviews commits, not individual edits, making the review cadence coarser but still human-driven. Both tools are designed for partial-horizon tasks where human judgment supplements model capability at decision boundaries.

**IDE agents — Cursor's agent mode, Windsurf** — embed the model into the editing surface, making feedback immediate and visual. Cursor's agent mode can perform multi-file edits and run terminal commands, but the interaction loop is built around the developer observing changes in real time and steering continuously. The latency tolerance is low (seconds per round-trip) and the task scope is typically bounded by a single feature or bug fix. This is the opposite of the factory model's $1,000/day token budget and multi-hour autonomous runs.

**Fully autonomous agents — Devin (Cognition Labs), OpenHands** — occupy a middle ground closer to Attractor. Devin provides a cloud-hosted environment where tasks are issued via a chat interface and the agent works autonomously, surfacing results when done. OpenHands (open-source, MIT-licensed) is model-agnostic and supports headless execution in pipelines, making it architecturally compatible with factory integration. Both differ from Attractor in that they are product surfaces with managed execution environments, whereas Attractor is an open specification — a pattern — that implementations instantiate using whatever model and infrastructure the factory controls.

**Amp and OpenAI Codex** sit between interactive CLI and fully autonomous. Amp is designed for longer autonomous runs but maintains human approval gates at key transitions. Codex CLI operates headlessly and composes with Unix pipelines, making it natural for CI use, but its loop design does not include the explicit loop detection, subagent decomposition, or provider-aligned tool routing that Attractor specifies.

The architectural distinction that matters most for factory deployment is not autonomy level but feedback architecture: interactive agents surface the model's uncertainty to a human for resolution; non-interactive agents must resolve uncertainty internally through tooling, retries, and scenario harnesses. Attractor-style agents are not more capable on any given step — they are designed to succeed statistically across many runs with zero human-hours per run, which changes the economics of software production fundamentally.

## Summary

The viability of long-horizon autonomous coding loops rests on a precise set of harness-level design choices that convert error signals into recovery opportunities rather than failure states. The coding agent loop must catch tool failures, detect repetitive patterns, enforce timeouts, align tools with model training, and emit structured events for observability. None of these are model capabilities — they are infrastructure choices. The outer harness must close the feedback loop with scenario-based holdout testing rather than model-generated test suites, because any test suite that the same agent generates can be overfitted to without producing correct behavior.

StrongDM's three named techniques — Gene Transfusion, Semport, and Pyramid Summaries — are each solutions to a class of problem that arises at factory scale regardless of the underlying models used. Gene Transfusion solves pattern propagation across a heterogeneous code estate without shared libraries. Semport solves cross-language synchronisation, enabling a factory to consume upstream innovation without adopting upstream technology choices. Pyramid Summaries solve context exhaustion, making enumeration and selective expansion tractable at codebase scales that exceed any context window. Together they constitute a portable vocabulary for factory infrastructure design that generalises beyond StrongDM's specific stack.

The non-interactive agent pattern represented by Attractor differs from interactive tools (Claude Code, Aider, Cursor) not in raw capability but in feedback architecture and economic model. Interactive agents amortise model uncertainty over human attention; non-interactive agents amortise it over computational volume. The former optimises for quality per human-hour; the latter for quality per dollar of token spend at zero human-hours per run. The factory paradigm is a bet that the latter curve is now steep enough — and model reliability high enough — to close the gap with human-reviewed development on a cost-adjusted basis.
