# Devlog — 2026-04-06

## Summary
Long session in three parts: (1) fixing the agent spawn pipeline end-to-end and reworking the bootstrap flow into a user-driven "hatch" experience, (2) setting up the devlog system itself and cleaning up stale planning docs, and (3) designing the memory layer — two independent research inputs (Karpathy's LLM Wiki PRD and a BEARY whitepaper) synthesized into a single implementation plan.

## Changes

### Spawn pipeline fixes
- **Read-only fs git config error** — `docker-entrypoint.sh` was running `git config --global` which tried to write `/root/.gitconfig` on the read-only root fs. Fixed by reordering: `git init` first, then local (non-`--global`) `git config`, so writes go to `/workspace/.git/config`.
- **Makefile `agent` target ignored `ID=`** — `make agent ID=foo` did nothing because the target hardcoded the script call. Now passes `$(ID)` through.

### MCP error handling
- Agent crashed with a scary `404 Not Found` traceback when the gateway didn't have the agent enrolled. Added two typed exceptions in `mcp/client.py`:
  - `MCPGatewayUnavailable` — gateway unreachable
  - `MCPAgentNotEnrolled` — agent missing from `agents.json`
- `_bootstrap_session` raises these instead of bare `httpx.HTTPStatusError`.
- `agent/graph.py` MCP loader now logs a clean `WARNING` for these expected cases and reserves `log.exception()` for unexpected failures. Agent continues without MCP tools.

### Hatch flow rework
- Removed automatic bootstrap. Previously `__main__.py` detected `BOOTSTRAP.md` and silently ran a hardcoded prompt with my name baked in.
- New flow: when `BOOTSTRAP.md` exists, `_chat_loop` collects user input interactively before the agent runs.
- Added `HatchIdentity` Pydantic model — collects `name` (required), `emoji`, `creature`, `vibe` (all optional but prompted).
- New `_write_identity_file()` pre-fills `IDENTITY.md` from the user's answers; bootstrap prompt now tells the agent **not to overwrite** it.
- Agent still handles the rest from `BOOTSTRAP.md` (USER.md, daily note, skills install, deleting BOOTSTRAP.md).
- Deleted dead code: `_run_bootstrap()` and `_invoke_graph()`.

### Dockerfile / build
- Embedding model wasn't actually cached in the image — `RUN python -c "...TextEmbedding(...)" 2>/dev/null || true` was silently swallowing failures, so every container spawn re-downloaded the model from HF (slow + breaks on read-only fs).
- Set `FASTEMBED_CACHE_PATH=/opt/fastembed_cache` and `HF_HOME=/opt/hf_cache` at build time so the model lands in a fixed location that's baked into the image layer.
- Removed `|| true` so the build fails loudly if the download fails.
- Discovered NeMo Guardrails also needs `sentence-transformers/all-MiniLM-L6-v2` (not just `BAAI/bge-small-en-v1.5`) — added both to the prefetch RUN.

### Memory layer design
This was the core of the second half of the session. Two distinct research inputs landed hours apart and had to be reconciled.

- **Whitepaper synthesis**: delegated an Explore agent to cross-reference all six existing whitepapers against the actual implementation. Result: ~77% of designs are implemented. The LangGraph runtime, MCP gateway, NeMo Guardrails, phantom-token auth, and Cerbos RBAC are all done. The largest gap is the memory layer — the `agent-memory-architecture` whitepaper proposed embedded LanceDB with hot/warm/cold tiers, none of which exists. Today memory is "read every vault markdown file into the system prompt on every invocation." No retrieval, no indexing, no distillation.
- **Confirmed intuition via GitHub issues**: open issue #3 on `FnSK4R17s/commandclaw` is literally "Design memory architecture" from 2026-03-29. The MCP work was a detour; memory was always next.
- **Considered the wrong framing first**: initially proposed `commandclaw-memory` as a separate repo/service and got pushback — "the vault IS the memory per VISION." Then the user reframed: yes, a separate service, but with vault as the cold/audit tier and the service as the performance layer. Vault and memory had been conflated.
- **Tried to articulate the fundamentals**: five irreducible primitives in any claw-shaped platform — loop runner, perception bus, effector layer, memory layer, substrate. Skills are a catalog format, guardrails are filters, tracing is an observer. None of those are primitives. Memory and substrate had been smushed together.
- **Launched a BEARY research pass** (`whitepaper-output/commandclaw-memory-service/`) in hyperphagia/unattended mode on a sonnet subagent. Asked it to challenge the five-primitive framing first, then solve the memory split. It ran ~15 minutes and came back validating the framing, recommending Option C (dedicated HTTP microservice with a LangGraph `BaseStore` shim), LanceDB for storage, phantom-token + HMAC + Cerbos auth matching the MCP pattern, and async distillation.
- **Karpathy collision**: while BEARY was running, the user dropped a [Karpathy gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) with a fundamentally different proposal — an LLM-maintained markdown wiki with `index.md`/`log.md`, no vector DB, no service. Saved it as [MEMORY_PRD_KARPATHY.md](../MEMORY_PRD_KARPATHY.md) in guiding_docs.
- **Initial attempt to phase them was rejected**: I proposed "Karpathy as v1, BEARY as v2." The user pushed back hard: this is enterprise software, Claude is expensive so we can't assume the primary agent is disciplined enough to maintain the wiki unaided, and both layers need to coexist. BEARY is the discipline and retrieval infrastructure; Karpathy is the data model and workflow contract. Neither gets archived.
- **Wrote the new [PLAN.md](../PLAN.md)**. 14 sections, ~670 lines. Treats the wiki pattern as the schema the memory service enforces. Every wiki write goes through a REST boundary with Pydantic validation so a cheap model can't corrupt the structure. Distillation runs async with a stronger model off the hot path to control cost. Build is phased 0–10 with exit criteria per phase. Signoff checklist at the bottom. Not to be implemented without explicit approval.

## Notes / observations
- Build took ~4 hours the first time on slow network (~16 kB/s). Suspect the MacBook on 5 GHz was hogging bandwidth. After switching networks, ~1 MB/s — 220 MB of models pulled in 214s.
- Layer caching strategy is paying off: after the first slow build, code-only changes only invalidate layer 2 and rebuild in seconds.
- BEARY's whitepaper and Karpathy's PRD agree on more than the first glance suggests: both land on "git vault = source of truth, LLM does the curation, cold tier is markdown, hot tier is small and in-prompt." The real disagreement is narrow: does warm retrieval need a vector index, or does a disciplined `index.md` suffice? For frontier Claude the answer is Karpathy. For cheaper models it's BEARY. Since CommandClaw has to support both, both layers stack.
- The corrected framing isn't adding complexity — it's removing a conflation. Before today, "memory" and "vault" were the same noun in the codebase. They shouldn't be.
- BEARY flagged "distillation quality" as the biggest unresolved risk. It is. Prompt engineering for fact extraction is the one thing no amount of research substitutes for.
